# Platform Runtime Plane

## Purpose

The runtime plane gives Musahama a shared execution model across modules.

Instead of every module inventing its own run-tracking vocabulary, the platform now standardizes around:

- cases
- runs
- run steps
- artifacts
- approval requests
- action executions

## Core Tables

The shared runtime plane is backed by:

- `module_cases`
- `module_case_entities`
- `module_runs`
- `module_run_steps`
- `module_artifacts`
- `module_action_requests`
- `module_action_executions`
- `module_evidence_links`

## Core Vocabulary

### Case

A case is the top-level unit of work for a module.

Examples:

- an origination search program
- a negotiation thread
- a compliance review
- a funding preparation effort

### Run

A run is one execution attempt inside a case.

Examples:

- a research execution
- a compliance scoring pass
- a funding-match generation pass

### Run step

A run step records bounded execution progress.

Examples:

- discover companies
- enrich evidence
- draft one-pager
- request approval

### Artifact

Artifacts are structured outputs generated by the platform.

Examples:

- one-pagers
- scorecards
- negotiation briefs
- compliance summaries
- funding recommendation packets

### Action request

An action request is a side-effectful operation waiting for approval.

Examples:

- send outreach
- write to CRM
- schedule a meeting
- submit an external application

### Action execution

An action execution is the actual performed outbound step after approval.

## Approval Model

The current platform assumes:

- internal analysis can run automatically
- outbound or side-effectful actions may require approval

This is why `Approvals` is now a first-class platform area instead of a feature hidden inside one module.

## Bridge Rule for Origination Match

`Origination Match` still uses legacy workflow tables for operational depth:

- `mandates`
- `research`
- `strategies`
- `research_results`
- `research_run_logs`

The bridge into the shared runtime plane currently happens through:

- `research.module_case_id`
- `research_run_logs.module_run_id`

This lets the old workflow continue operating while the platform moves toward a shared module runtime model.
